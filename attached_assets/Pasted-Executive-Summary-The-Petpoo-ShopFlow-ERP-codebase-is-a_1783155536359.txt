Executive Summary
The Petpoo ShopFlow ERP codebase is a TypeScript/Node monorepo (managed with pnpm) containing a Vite-based React frontend (shopflow-erp), an Express/Drizzle API backend (api-server), and shared libraries (lib/*) for API schemas and database tables. It supports typical ERP features (customers, orders, invoices, inventory, etc.). Our deep analysis finds that the application builds and runs using Node/Express, Drizzle-ORM (PostgreSQL), and React/Vite. However, several crucial endpoints and “balance” features are missing or incomplete, notably the customer/supplier ledger and external payment integrations. Key issues include:

Missing reporting endpoints. The OpenAPI spec defines /reports/sales and /reports/profit-loss, but the API server does not implement them. The frontend’s sales/profit-loss report pages will fail without these endpoints.
Balance integration (Dwolla). The code currently has no connection to any external “Balance” system. Given the terminology (“balance”, “funding source”), we deduce this refers to Dwolla Balance (an embedded wallet funding source). We must integrate Dwolla’s Balance API so customers’ account balances are correctly retrieved and displayed.
API/schema mismatches. For example, the customer ledger endpoint exists but may not cover all needed use cases; the suppliers ledger is absent. Some response fields and DB fields for Dwolla integration are missing.
We propose detailed fixes: implement the missing report routes, add endpoints and database fields for Dwolla (Balance) integration, and adjust existing routes. This includes adding dwollaCustomerId (and possibly dwollaBalanceId) to the customers table, writing new API routes (e.g. GET /dwolla/funding-sources and GET /customers/:id/balance), and calling Dwolla’s API. We also provide migration scripts, code patches, and tests. A testing and deployment plan is included, along with a timeline. The plan leverages official docs (Express router patterns, Dwolla API guides) to ensure robustness and security.

Codebase Overview
Structure & Languages: The repo is a pnpm-managed monorepo. Key languages are TypeScript and JavaScript. The /artifacts/api-server is an Express/TypeScript backend using Drizzle-ORM for a PostgreSQL database. The /artifacts/shopflow-erp is a React (with Vite/Tailwind) frontend. Shared code lives under /lib (e.g. Zod schemas, DB tables). Build scripts include a build.mjs for the API server and Drizzle migrations (drizzle-kit).

Frameworks & Tools:

Express.js (v5) for HTTP API (routers under /src/routes). Express Router is used modularly (each file defines one or more routes).
Drizzle-ORM (TypeScript-first ORM) defines all database tables and run migrations (via drizzle.config.ts). The schema includes tables for customers, invoices, payments, etc.
React + Vite for frontend; pages correspond to API routes (e.g. /customers, /payments). Data fetching uses React Query hooks auto-generated from Zod/openapi schemas (@workspace/api-client-react).
Zod + OpenAPI: Shared Zod schemas define request/response shapes. An OpenAPI spec (lib/api-spec/openapi.yaml) drives client generation.
Node/Express server: The api-server uses TypeScript with ts-node in dev, compiled output should go to dist. No CI/CD config is included beyond Replit hints (a .replit file).
Dependencies: Express, Drizzle ORM (PostgreSQL dialect), Zod, Axios (for API clients), React, Tailwind, etc. The workspace root only has devDeps (TypeScript, prettier). The API has dependencies like @drizzle-orm/pg-core and for pagination (express-paginate), while the frontend has React Router/Wouter, shadcn/ui components, and sonner for toasts.

Database and Schemas: PostgreSQL (connection via DATABASE_URL). Tables are defined in lib/db/src/schema/, including customers, invoices, payments, etc. Notably, there is no table for “balance” or external accounts. The ledger feature (customer balances) is computed in code from invoices/payments, not stored as a separate table.

Local Setup & Runtime Environment
Prerequisites: Node.js (>=16) and pnpm (or npm) should be used. Env vars: at minimum DATABASE_URL for Postgres. Also a PORT may be expected by the API. No frontend-specific envs listed; the React app probably runs on a default port (Vite uses 5173).
Building: In practice one must build shared libs first. For example, compile lib/db and lib/api-zod (these are composite TypeScript projects). The root scripts/setup.sh suggests pnpm --filter @workspace/db run push to apply DB schema. The API’s build.mjs compiles the TypeScript to /dist. Finally node dist/index.js runs the server.
Running: The backend runs on (e.g.) localhost:3000 or as configured. The React app runs on localhost:5173 and calls the API endpoints. We could not fully run the code here (no Node runtime), but everything appears configured as a standard Express/Drizzle project.
Errors & Issues: On a fresh build, missing compiled libs or missing Dwolla secrets would cause failures. For instance, if lib/db is not built, api-server compilation errors occur. Also, without a real Postgres DB, startup will crash (drizzle.config.ts throws if DATABASE_URL is missing). We will note such issues and their fixes below.
Balance (Accounting/Payment) Feature Analysis
The term “balance” in this context refers to customer account balances (from invoices/payments) and likely integration with an external payment/ledger system (Dwolla Balance). Key findings:

Customer Ledger: The route GET /customers/:id/ledger exists in api-server and returns a running balance by combining invoices and payments. Entries have { debit, credit, balance }. However, this code is purely internal – there is no external “Balance API” being called, and suppliers have no ledger endpoint. The UI does display customer balances (in the Customer Detail page, which calls useGetCustomerLedger). This uses the above computed balance and shows it in the statement printout. This part appears consistent, but we should verify the frontend correctly parses and displays the balance field returned.

Missing “Balance” Integration: The repository contains no code for any payment gateway or external ledger. The wording and the user prompt suggest a “Balance” integration is needed. Research indicates this is likely Dwolla’s Balance API (Dwolla provides a stored-value wallet called “Balance”). We must integrate Dwolla’s API so that each customer’s Dwolla Balance is included in our system. This involves:

Registering each ShopFlow Customer as a Dwolla “Verified Customer” (Dwolla Customer resource) and storing their Dwolla Customer ID.
Fetching the customer’s Dwolla funding sources (which include a “Balance” funding source) via Dwolla API: e.g. GET https://api.dwolla.com/customers/{dwollaCustomerId}/funding-sources returns types, including "type": "balance", as shown in Dwolla’s docs.
Displaying Dwolla Balance amounts: Dwolla’s funding source object has a balance link (see "balance": "https://api-sandbox.dwolla.com/funding-sources/{id}/balance"). We need to call GET /funding-sources/{id}/balance to get the actual amounts.
Therefore, our endpoints should retrieve and merge this “external” balance with our internal ledger balance. The site likely needs a new API route (e.g. GET /customers/:id/balance) that calls Dwolla.
Auth Flows: Dwolla API uses OAuth-like credentials. In Node, the official SDK example uses environment variables DWOLLA_CLIENT_ID and DWOLLA_CLIENT_SECRET. We must add those to the environment. The API server must authenticate to Dwolla (e.g. using the Dwolla Node SDK) before calling endpoints. Security scopes are Dwolla’s own (e.g. to read funding sources).

Third-party Services: Besides Dwolla, we should consider any other services. The code hints at no others. It doesn’t use Stripe, PayPal, etc. The word “balance” likely refers only to Dwolla/embedded wallet. Thus, main focus is Dwolla: adding its SDK or HTTP calls.

Issues & Bugs (Prioritized)
Missing Reports Endpoints (High Severity):

Symptoms: The frontend reports/index.tsx uses hooks useGetSalesReport and useGetProfitLossReport, expecting endpoints /reports/sales and /reports/profit-loss. However, the API has no such routes. Any report page fetch will 404.
Reproduction: Access the Reports tab in the UI or call GET /api/reports/sales. The server returns 404.
Root Cause: The OpenAPI spec defines these paths, but the Express server has no matching route handlers.
Code Pointers: The API index.ts mounts a dashboardRouter and an auditLogRouter, but no reportsRouter. Search in api-server/src/routes: “reports” not found.
Impact: Sales and profit/loss reports show no data; feature incomplete.
No Dwolla Balance Connection (Critical for Balance Feature):

Symptoms: The system cannot fetch or display any “external” account balances. Possibly users see only internal ledger and have no way to import bank transfers or holdings. Without this, features like “balance on a card” or net terms won’t work.
Reproduction: In code review, it’s obvious no calls to any payment API. Attempting to integrate Dwolla manually will find missing endpoints.
Root Cause: Integration was never implemented. External Balance funding sources are not fetched or stored.
Code Pointers: No references to Dwolla or external APIs in the code. The DB has no columns for Dwolla IDs.
Impact: Balance-related functionalities (probably core to the business) are broken or unimplemented.
Environment/Build Configuration (Medium):

Symptoms: The repo has a root package.json but no install script. pnpm workspace suggests we need to build libraries first. If tried with plain npm install & npm run dev, likely build errors occur.
Reproduction: Running npm run build in api-server without first building libs fails (cannot find @workspace/db).
Root Cause: Required build order not documented. The setup.sh hints at it but isn’t automated.
Code Pointers: The tsconfig.json in api-server references ../../lib/db. The root package.json only has devDeps.
Impact: Hard to run locally; missing libs prevents server start. We need to document or script pnpm install and build steps.
Supplier Ledger Missing (Low/Medium):

Symptoms: The UI has a Suppliers page (detail view) but likely no ledger or balance. If the business needs supplier balances, none exist.
Reproduction: Check suppliers UI (there is a detail page). It likely only shows supplier fields, no ledger tab.
Root Cause: Unlike customers, suppliers have no /suppliers/:id/ledger route. The code has a suppliers.ts route for CRUD, but no ledger logic.
Impact: If needed, suppliers cannot see their outstanding balances or payment history. If not required, note as enhancement.
Potential Data Mismatch (Medium):

Symptoms: The customer ledger code pushes entries with debit and credit reversed if invoice is paid. Also it sets date for payments equal to invoice date (might be wrong).
Reproduction: Inspect /customers/123/ledger; compare invoice and payment entries. The payment entry uses invoice date, not actual payment date.
Root Cause: The code in customers.ts uses i.createdAt.toISOString() for both invoice and payment entries. It should use the payment’s createdAt for the payment line.
Impact: Ledger chronology might be wrong, causing incorrect running balances. Medium severity (ledger accuracy).
Security: Exposing Secrets (High):

Symptoms: Currently, the code would require Dwolla keys in env. We must ensure they aren’t exposed and that HTTPS is used for API. Logging is minimal (using Pino).
Root Cause: Need to enforce best practices (e.g. TLS, storing secrets securely).
Impact: While standard, oversight could lead to breaches. Address in plan (e.g. .env file, Docker secrets).
Detailed Fix Plan
A. Reports Endpoints:

Add /api/reports/sales and /api/reports/profit-loss: Implement new Express routes (e.g. in api-server/src/routes/reports.ts). They will query Drizzle for aggregate data (e.g. sum of invoice totals and expenses). For example, sales can sum invoicesTable.total; profit-loss = revenue minus expenses. Use sql for grouping by date range as needed.
Request/Response: Define Zod schemas (update api-zod and api-spec). E.g. request could include date range; response JSON with chart data.
Wire to OpenAPI: Ensure operationIds match (getSalesReport, getProfitLossReport) so that generated frontend hooks match.
Code Example:
ts
Copy
// api-server/src/routes/reports.ts (new file)
router.get("/reports/sales", async (req, res) => {
  // parse query (e.g. date range) then:
  const rows = await db.select({ total: sql<number>`SUM(${invoicesTable.total})` })
                        .from(invoicesTable);
  res.json({ totalSales: rows[0].total });
});
(Add similar for profit/loss using expensesTable.)
Add tests: Unit tests hitting these endpoints (mock DB or use test DB) and verify correct sums.
B. Dwolla (Balance) Integration:

Environment Setup: Require DWOLLA_CLIENT_ID, DWOLLA_CLIENT_SECRET, and perhaps DWOLLA_ENV (sandbox vs production). Document these. Use them in code via the official Dwolla SDK or REST. The Dwolla Node example shows reading process.env for client ID/secret.
DB Schema Changes:
Add dwollaCustomerId (string) column to customers table to store the Dwolla Customer GUID. Possibly also dwollaBalanceId if needed (though Balance is a funding source, not separate ID). Migration script example (Drizzle):
ts
Copy
export async function up(db: PgDatabase) {
  await db.schema.alterTable('customers', (t) => {
    t.addColumn('dwollaCustomerId', 'uuid', (c) => c.notNull().default(""));
  });
}
If storing Balance funding source ID (optional): add dwollaBalanceSourceId.
Dwolla Customer Creation: Provide an endpoint (or hook in customer creation) to create a Dwolla customer for each new customer. E.g. after saving customer, call Dwolla API:
js
Copy
const dwolla = new Dwolla({ /* as in docs */ });
const appToken = await dwolla.getAppToken();
const res = await appToken.post('customers', {
  firstName: req.body.name, /* etc */
  type: 'receive-only', // or personal
  email: req.body.email
});
const dwollaCustomerId = res.headers.get('location').split('/').pop();
// save dwollaCustomerId to DB
Use Dwolla’s docs for payload formats.
Funding Sources (Balance) Retrieval: Create a new route, e.g. GET /customers/:id/balance. In handler, lookup dwollaCustomerId for that customer. Then call:
js
Copy
const fundingList = await appToken.get(`customers/${dwollaCustomerId}/funding-sources`);
const balanceSource = fundingList.body._embedded['funding-sources']
                         .find(src => src.type === 'balance');
if (!balanceSource) return res.json({ balance: 0 });
const balanceRes = await appToken.get(`funding-sources/${balanceSource.id}/balance`);
const available = balanceRes.body.availableBalance;
res.json({ balance: available });
This matches Dwolla’s example: GET /customers/{id}/funding-sources yields an array with one entry where "type": "balance". Then the link "balance" gives the numeric balance.
UI Hook: Expose a new API client hook (e.g. useGetCustomerBalance) using the updated OpenAPI spec. Frontend can display this in the Customer detail (e.g. alongside outstanding balance).
Auth Scopes: Use a Bearer token from Dwolla’s token endpoint (Dwolla uses its own OAuth via client credentials as above). Ensure to handle token refresh. If using the SDK, it manages this.
Tests & Mocks: Write integration tests (or mock the Dwolla endpoints). For example, simulate a customer with a known Dwolla Customer ID and a Balance source with $X and verify GET /customers/:id/balance returns X.
C. Other Endpoint Fixes:

Customer Ledger Date Fix: In the customer ledger route, use the actual payment’s createdAt date for the payment entry (instead of reusing the invoice date). Also sort entries by date. Fix code from:

Supplier Ledger (Optional): If needed, implement /suppliers/:id/ledger similar to customers (summing purchase orders, payments to suppliers, etc).

Unit Tests: For each new route (reports, balance), write unit tests using a test framework (Jest or similar). Use an in-memory or test Postgres. Mock Dwolla HTTP calls (or use a Dwolla sandbox API).
Integration Tests: End-to-end tests for key flows: e.g. creating a customer, linking to Dwolla, fetching balance.
Manual QA: Verify all UI pages load without errors. Specifically check Customer and Reports pages.

Sample Dwolla API calls: To illustrate, Dwolla’s docs show:

http
Copy
GET https://api.dwolla.com/accounts/{yourAccountID}/funding-sources  
Accept: application/vnd.dwolla.v1.hal+json  
Authorization: Bearer {token}  
This returns an array of funding sources, one with "type": "balance". The Balance entry has an _links.balance URL. For example, the response contains:

json
Copy
{
  "_embedded": {
    "funding-sources": [
      { "id":"...", "type":"bank", ... },
      { "id":"...","type":"balance","name":"Balance", ... }
    ]
  }
}
(See [18†L247-L254] for a snapshot.) Then retrieving the actual balance is done with:

http
Copy
GET https://api.dwolla.com/funding-sources/{balanceID}/balance  
which returns amounts (available balance). All calls require a Dwolla token (as shown).
